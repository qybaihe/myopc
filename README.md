# MyOPC

**MyOPC 是面向一人公司、独立开发者和小型 AI 团队的全栈运营系统。**

它不是一个单点工具，而是一套把 **想法、产品、代码、部署、支付、通知、监控、数据和 AI 员工** 串起来的操作系统。用户不需要在十几个平台之间切换，不需要自己拼接 agent、域名、服务器、支付、消息、知识库和监控链路。MyOPC 的目标是让一个人也能拥有一家成熟互联网公司的完整数字基础设施。

> 一句话愿景：让用户从“我有一个想法”一路走到“产品上线、用户付款、系统自动运营、AI 团队持续迭代”。

---

## 产品愿景

MyOPC 想做的是 **AI-native Company OS**。

今天的个人创业者通常要同时面对这些问题：

- 产品怎么搭建，代码谁来写，需求怎么拆？
- 域名、二级域名、HTTPS、部署、回滚、监控怎么处理？
- 用户注册、邀请、权限、团队协作怎么打通？
- 支付、订阅、订单、退款、发票、收入看板怎么接？
- 线上挂了谁提醒，支付回调失败谁处理，用户反馈谁跟进？
- AI agent 做了什么、下一步该做什么、预算花了多少、结果是否可靠？

MyOPC 把这些问题收束到一个工作台里：**每个项目都是一个可运营的产品，每个 AI 员工都有角色和任务，每个事件都能被通知、追踪和处理，每个上线动作都能进入收入和数据闭环。**

## 面向用户的前端体验

MyOPC 前端不是工具拼盘，而是一个统一的产品界面。

用户看到的是 MyOPC 自己的工作台，而不是底层开源项目、命令行或服务器面板。底层能力会被抽象成清晰的业务模块：

- **仪表盘**：一眼看到收入、访客、部署、告警、AI 员工状态和今日待处理事项。
- **项目中心**：每个项目都有域名、代码、部署、支付、数据、知识库和驻守 AI 员工。
- **AI 员工**：招聘、配置、分配任务、查看进度、审阅输出、控制预算。
- **任务与审批**：所有代码修改、部署、支付配置、内容发布都可以进入审批流。
- **通知中心**：该提醒你的事情一定出现，不该打扰你的事情自动归档。
- **支付与收入**：订单、订阅、支付回调、退款、收入趋势和转化漏斗集中管理。
- **域名与发布**：一键生成二级域名，绑定自定义域名，自动 HTTPS，自动路由。
- **监控与分析**：在线状态、接口可用性、支付回调、页面转化和用户行为统一呈现。
- **知识库**：项目 SOP、需求、会议记录、架构决策、运营手册都能被 AI 员工直接调用。

设计原则是：**复杂能力在后台流动，用户界面只保留清晰、稳定、可执行的决策入口。**

## 核心能力

### 1. 多 Agent 管理系统

MyOPC 把 AI agent 当作公司里的数字员工来管理，而不是当作一次性聊天窗口。

- 每个 agent 有角色、职责、预算、技能、上下文和项目归属。
- 任务可以被拆分、排队、分配、继续、暂停、审批和复盘。
- Agent 可以负责开发、运维、内容、增长、客服、数据分析和财务运营。
- 用户可以看到 agent 正在做什么、为什么这么做、下一步需要谁批准。
- Agent 输出不会直接失控上线，关键动作会进入审批或安全策略。

### 2. 编码与产品迭代引擎

MyOPC 内置智能编码系统，用于真实推动产品开发，而不是只给建议。

- 读取项目上下文、知识库、任务说明和历史决策。
- 自动修改代码、运行检查、总结变更、提出下一步。
- 支持按项目绑定工作区，让每个产品都有自己的开发执行环境。
- 通过统一 AI 网关接入 MiniMax M3 等模型，减少模型切换和配置摩擦。
- 用户前端只看到 MyOPC Code Engine，底层执行细节由系统托管。

### 3. 通知、提醒与待办中心

MyOPC 的通知不是简单消息列表，而是业务事件的指挥塔。

系统会在这些场景主动提醒用户：

- 有新用户注册、邀请接受、权限变更。
- 支付成功、支付失败、订阅即将续费、退款需要处理。
- 域名解析异常、证书即将过期、站点不可用。
- 部署完成、部署失败、回滚完成。
- AI 员工请求审批、任务卡住、预算接近上限。
- 用户反馈进入、数据异常波动、转化率突然下降。

提醒目标是：**重要事件不漏掉，重复噪音不打扰，所有通知都能变成可执行动作。**

### 4. 支付与商业化闭环

MyOPC 要把所有常见支付场景打通，让一个项目从第一天起就是可收费的产品。

目标覆盖：

- 一次性付款、订阅、套餐升级、降级、试用期。
- 优惠码、订单、退款、失败重试、支付回调。
- 用户权限和支付状态联动，例如付款后自动开通功能。
- 收入看板、项目收入、用户生命周期价值、转化漏斗。
- 支付异常通知和自动排查，例如 webhook 失败后创建运维任务。
- 多项目、多产品、多域名下的收入归集。

最终用户不需要手写一套商业化后台。MyOPC 会把支付配置、产品权限、通知和数据分析连成一个闭环。

### 5. 二级域名分发与产品上线

MyOPC 的每个项目都应该可以快速变成一个可访问、可监控、可收费的线上产品。

目标能力：

- 自动生成 `<project>.apps.myopc.me` 这类二级域名。
- 支持绑定自定义域名和多租户子域名。
- 自动配置 HTTPS、反向代理、路由和健康检查。
- 部署后自动进入监控和数据分析。
- 支持回滚、重新部署、环境变量管理和上线审批。
- 让 AI 员工可以在授权范围内完成部署、检查和修复。

这意味着 MyOPC 不只是写代码，也负责把代码送到用户面前。

### 6. 一站式项目运营

MyOPC 的长期目标是覆盖一个产品从 0 到 1 再到持续运营的完整生命周期：

1. 创建项目：输入想法、目标用户、商业模式。
2. 生成计划：拆解需求、页面、接口、数据库、支付和部署路径。
3. 分配员工：让开发、运维、内容、增长 agent 接手不同任务。
4. 构建产品：代码修改、知识沉淀、测试检查、版本记录。
5. 上线发布：域名、HTTPS、部署、监控、分析自动配置。
6. 接入支付：商品、订阅、订单、权限和回调打通。
7. 运营增长：通知、数据、用户反馈、转化优化、自动迭代。

## 技术方向

MyOPC 采用开放式集成思路：前端体验统一，底层能力模块化。

- **Frontend**：项目级工作台，负责将复杂能力重绘成统一的 MyOPC UI。
- **Control Plane**：管理公司、项目、AI 员工、任务、审批、预算和知识。
- **Code Engine**：负责代码执行、项目迭代、检查和开发任务推进。
- **AI Gateway**：统一模型接入，当前重点接入 MiniMax M3。
- **Notification Layer**：把支付、部署、监控、审批、用户行为变成可处理事件。
- **Commerce Layer**：打通订单、订阅、支付回调和产品权限。
- **Domain & Deploy Layer**：处理二级域名、HTTPS、反向代理、部署和回滚。
- **Observability Layer**：站点状态、接口状态、支付回调、访问分析和异常追踪。

## 安全与密钥

MyOPC 不把真实 API Key、支付密钥、邮件密钥或服务器凭据写进仓库。

推荐做法：

- 使用 `.env` 或服务器私有环境变量保存密钥。
- 示例文件只保留空值和占位说明。
- 提交前做 secret scan。
- GitHub Secret Scanning alert 需要及时处理、关闭或轮换相关密钥。
- 支付、AI 网关、邮件发送、部署密钥都必须通过服务端注入，不在前端暴露。

## 路线图

### Phase 1：统一工作台

- 注册、登录、公司自动创建。
- 项目、任务、AI 员工、审批和知识库基础闭环。
- MyOPC Code Engine 默认接入 MiniMax M3。
- 前端品牌统一为 MyOPC。

### Phase 2：事件与通知

- 通知中心。
- 支付、部署、监控、审批事件统一进入 Inbox。
- 邮件、站内提醒、Webhook 多通道通知。
- 重要事件自动生成可执行任务。

### Phase 3：支付与商业化

- 商品、订单、订阅、退款、优惠码。
- 支付回调监控和失败重试。
- 用户权限与支付状态联动。
- 收入看板和项目级商业指标。

### Phase 4：域名与部署分发

- 项目二级域名自动分发。
- 自定义域名绑定。
- HTTPS、反向代理、健康检查。
- 一键部署、回滚、环境变量管理。

### Phase 5：一人公司自动运营

- AI 员工驻守项目。
- 根据告警、收入、用户反馈和数据波动自动提出行动。
- 项目从需求到上线再到增长的持续自动化。
- MyOPC 成为个人和小团队的完整 AI 公司操作系统。

## 终局

MyOPC 的终局不是“又一个 AI 工具”，而是一个真正能承载业务的系统。

它应该让用户感觉自己不是在管理软件，而是在管理一家由 AI 员工、自动化基础设施和商业闭环组成的公司。

**一个人，也可以拥有产品团队、工程团队、运维团队、增长团队、支付系统、通知系统和全球发布能力。**

这就是 MyOPC。
