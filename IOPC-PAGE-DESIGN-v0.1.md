# IOPC 页面设计规格 v0.1

## 核心设计原则

### 原则一：IOPC 是唯一的产品，子项目只是数据源

```
❌ 错误做法                        ✅ 正确做法
┌────────────────────┐            ┌────────────────────┐
│  iframe: Kuma      │            │  IOPC 组件: 状态卡片 │
│  iframe: Umami     │            │  数据来自 Kuma API  │
│  iframe: 外部服务   │            │  数据来自 Umami API │
│  iframe: Traefik   │            │  用 IOPC Design Kit │
│  各自风格，拼盘感    │            │  渲染，统一体验      │
└────────────────────┘            └────────────────────┘
```

**铁律：用户永远不看到子项目的原生 UI。** 所有数据经 Adapter → IOPC Backend → IOPC 前端重绘。没有任何例外——包括文档编辑器在内，全部使用 IOPC 自绘组件。文档编辑用 Milkdown（React 组件）嵌入，而非 iframe 引入外部服务。

### 原则二：Paperclip 是中枢，页面围绕"人"组织

用户来 IOPC 是管理自己的"一人公司"，不是管理 7 个工具。页面按**业务对象**组织（员工、项目、Skill），不按**技术模块**组织（Kuma 面板、Traefik 面板）。

### 原则三：渐进式信息密度

- 首层：一句话 + 状态指示器（绿/黄/红点）
- 二层：关键指标卡片（数字 + 趋势小图）
- 三层：完整数据表 / 详情页

用户扫一眼首页就知道一切 OK 还是出事了，不需要读完整个页面。

## 统一渲染策略：哪些重绘，哪些嵌入

| 子项目 | 策略 | 理由 |
|--------|------|------|
| **Uptime Kuma** | 纯 API → IOPC 重绘 | 它的后台面板是给运维看的，不适合创业者 |
| **Umami** | 纯 API → IOPC 重绘 | 同上，而且 Umami 面板风格与 IOPC 完全不同 |
| **Traefik** | 纯 API → IOPC 重绘 | Traefik Dashboard 技术感太重，不适合 |
| **LiteLLM** | 纯 API → IOPC 重绘 | LiteLLM UI 是管理面板，用户不需要看 |
| **Paperclip** | 纯 API → IOPC 重绘 | Paperclip 是中枢，但 UI 按我们的"员工"隐喻重绘 |
| **Novu** | 纯 API → IOPC 重绘 | 只需读取通知列表和配置通道 |
| **Milkdown** | **React 组件嵌入 IOPC** | 编辑器本身就是 UI 组件，不是独立服务，天然融入 |
| **R2R** | 纯 API → IOPC 重绘 | R2R 没有面向终端用户的 UI |
| **OpenCode** | 不在 UI 出现 | OpenCode 是后端执行器，用户通过 Paperclip 间接调用 |

**Milkdown 是唯一以组件方式嵌入的，不是 iframe，不是独立服务。** 它是 npm 包，IOPC 的 React 组件树原生包含它，主题/插件/交互全部由 IOPC 控制。
